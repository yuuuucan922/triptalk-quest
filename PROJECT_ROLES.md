# Codex チーム運用手順書

## プロジェクト概要

- リポジトリ: https://github.com/yuuuucan922/triptalk-quest
- デプロイブランチ: main
- 公開URL: https://yuuuucan922.github.io/triptalk-quest/
- メインファイル: app.js / index.html

## 結論

このプロジェクトは、1つのチャットで全部決めるより、担当を分けて進める。  
ただし、コードを直接編集する担当は原則1つにする。

最も安定する進め方はこれ。

```text
企画・ゲーム設計
-> 英語学習コンテンツ
-> UI/UX・デザイン
-> 実装
-> レビュー
```

## 目的

- 思いつきを整理してから実装する
- 英語学習の質を上げる
- UIとゲーム性を分けて考える
- 今後のアップデートをしやすくする
- 変更がぶつからないようにする

## マルチエージェント運用の前提

この手順書でのマルチエージェントは、複数の担当を同時に自動実行する方式ではない。  
ユーザーがPMとして進行し、Codexに担当を指定して役割を切り替える。

基本形:

```text
ユーザーが目的を伝える
-> 企画・ゲーム設計担当が案を出す
-> ユーザーが採用案を決める
-> 英語学習コンテンツ担当が問題を作る
-> UI/UX・デザイン担当が画面案を確認する
-> 実装担当がファイルを編集する
-> レビュー・QA担当が確認する
-> ユーザーが完了判断する
```

重要ルール:

- 1回の依頼では、担当を1つだけ指定する
- 実装担当以外はファイルを編集しない
- 実装担当は、採用済みの仕様だけ反映する
- レビュー・QA担当は、原則として修正せず指摘だけ出す
- ユーザーが「採用」「保留」「却下」を決める
- 判断が曖昧なまま実装に進まない

## 担当別の専用プロンプト

各担当を呼び出すときは、最初に次の文を入れる。

### 企画・ゲーム設計担当

```text
企画・ゲーム設計担当として考えて。
まだファイル編集はしないで。
```

### 英語学習コンテンツ担当

```text
英語学習コンテンツ担当として考えて。
まだファイル編集はしないで。
```

### UI/UX・デザイン担当

```text
UI/UX・デザイン担当としてレビューして。
まだファイル編集はしないで。
```

### 実装担当

```text
実装担当として、採用済みの仕様だけ反映して。
必要なファイルだけ編集して。
```

### レビュー・QA担当

```text
レビュー・QA担当として確認して。
修正はまだしないで、指摘だけ出して。
```

## 成果物の受け渡しフォーマット

各担当は、最後に次の形式で次の担当へ渡す。

```text
## 担当
企画・ゲーム設計 / 英語学習コンテンツ / UI/UX / 実装 / レビュー・QA

## 結論
今回の結論を1〜3行で書く。

## 提案または変更内容
- ...
- ...
- ...

## 採用してよい項目
- ...
- ...

## 保留した方がよい項目
- ...
- ...

## 次の担当への引き継ぎ
次に誰へ渡すべきか、何を依頼すべきかを書く。
```

## 採用判断ルール

実装前に、ユーザーが各案を次の3つに分ける。

```text
採用: 今回実装する
保留: 後で検討する
却下: 今回はやらない
```

採用条件:

- 今フェーズの目的に合っている
- カフェエリアの完成に近づく
- スマホで使いやすい
- 英語学習とゲーム報酬がつながる
- 1回の実装で終わる大きさに収まる

保留条件:

- 面白いが今フェーズには大きすぎる
- オンライン機能が必要
- 画像素材や外部AIが必要
- 他エリア拡張後の方が自然

却下条件:

- 英語学習と関係が薄い
- 実装が重すぎる
- 夫婦で使う時にストレスになる
- 既存のカフェ体験を分かりにくくする

## ファイル編集権限

担当ごとの編集権限は次の通り。

| 担当 | ファイル編集 | できること |
|------|--------------|------------|
| 企画・ゲーム設計 | 不可 | 案出し、仕様整理、優先順位付け |
| 英語学習コンテンツ | 不可 | 問題案、解説案、難易度調整案 |
| UI/UX・デザイン | 不可 | 画面レビュー、改善案、導線整理 |
| 実装 | 可 | HTML/CSS/JS/Markdownの編集、構文チェック |
| レビュー・QA | 原則不可 | バグ指摘、完了条件確認、改善提案 |

例外:

- ユーザーが明示した場合だけ、レビュー・QA担当が軽微な修正まで行ってよい
- 企画・英語・UI担当は、コード例を出してもファイルには反映しない

## レビュー担当チェックリスト

レビュー・QA担当は、最低限これを確認する。

- 目的に合った変更になっているか
- 正解前に不要なヒントが出ていないか
- 正解後の解説が学習に役立つか
- XPや報酬が増えすぎないか
- スマホ幅で操作しにくくないか
- 既存の単語・英文・英会話モードが壊れていないか
- localStorageの既存データを壊していないか
- GitHub Pagesで表示できる前提になっているか
- 「1機能の完了条件」を満たしているか

## 進捗テーブル更新ルール

「現在の進捗」テーブルは、各担当フェーズが終わったら更新する。

更新基準:

- 企画: 採用案が決まったら「済」
- コンテンツ: 問題・解説・正解条件が決まったら「済」
- UI: スマホ前提の画面方針が確認できたら「済」
- 実装: ファイル編集と構文チェックが終わったら「済」
- レビュー: レビュー・QA担当が確認したら「済」

注意:

- 実装だけ終わっても、レビューは「未」のままにする
- 次フェーズに入る前に、カフェ行をできるだけ「済」に近づける

## 担当一覧

## 1. プロダクトオーナー

主担当: ユーザー

決めること:

- どんなアプリにしたいか
- どの案を採用するか
- 奥さんとどう遊びたいか
- どの機能を先に作るか

Codexに任せること:

- 案の整理
- 実装手順の作成
- 仕様書化
- コード実装

## 2. 企画・ゲーム設計担当

役割:

- エリア設計
- XP設計
- 報酬設計
- キャラクター解放
- 内装アップグレード
- 協力プレイ設計

見るポイント:

- 学習と報酬がつながっているか
- 毎日少し触りたくなるか
- 夫婦で一緒に遊ぶ理由があるか
- 複雑すぎないか

## 3. 英語学習コンテンツ担当

役割:

- 単語問題を作る
- 英文問題を作る
- 英会話ロールプレイを作る
- 難易度を調整する
- 正解後の解説を作る
- 復習設計を考える

見るポイント:

- 旅行で本当に使う表現か
- 初心者に難しすぎないか
- 簡単すぎて作業になっていないか
- 日本語ヒントを出すタイミングが適切か

## 4. UI/UX・デザイン担当

役割:

- スマホで見やすい画面を考える
- ボタン配置を考える
- カフェ、キャラ、内装の見せ方を考える
- 成長演出を考える
- 画面の情報量を調整する

見るポイント:

- スマホで押しやすいか
- 文字が小さすぎないか
- 何をすればいいか直感的に分かるか
- ゲーム画面として楽しいか
- 英語学習画面として邪魔になっていないか

## 5. 実装担当

役割:

- HTML/CSS/JSを編集する
- GitHub Pagesに反映する
- バグを直す
- データ構造を整理する
- 既存機能を壊さないように変更する

見るポイント:

- 変更範囲が広がりすぎていないか
- スマホで崩れないか
- GitHub Pagesで動くか
- localStorageの保存形式を壊していないか
- push前に構文チェックしているか

## 6. レビュー・QA担当

役割:

- 変更後に問題がないか確認する
- スマホ表示を想定して確認する
- 英語問題の難易度を見る
- ゲームバランスを見る
- バグや違和感を指摘する

見るポイント:

- 画面が崩れていないか
- XPが増えすぎないか
- 問題が簡単すぎないか
- 既存の機能が壊れていないか
- 奥さんに共有しても迷わないか

## 基本の進行手順

## Step 1: アイデア出し

まず実装しない。  
企画・ゲーム設計担当として案を出す。

例:

```text
カフェエリアをもっと楽しくしたい。
企画・ゲーム設計担当として、報酬や成長要素を10個提案して。
```

## Step 2: 採用案を決める

出た案から、ユーザーが採用するものを決める。  
全部入れない。

決める基準:

- 今すぐ作れる
- 学習とつながっている
- スマホで分かりやすい
- 奥さんと共有したとき楽しい

## Step 3: 学習内容を作る

英語学習コンテンツ担当に、問題と解説を作らせる。

例:

```text
英語担当として、採用したカフェ機能に合う問題を作って。
単語、英文、英会話に分けて。
```

## Step 4: UI案を作る

UI/UX担当に、画面配置や見せ方を考えさせる。

例:

```text
UI担当として、スマホで見やすい画面構成を提案して。
```

## Step 5: 実装する

仕様が決まってから、実装担当として反映する。

例:

```text
実装担当として、決定した仕様だけを反映して。
実装後に構文チェックして、GitHubへpushして。
```

## Step 6: レビューする

最後にレビュー・QA担当として確認する。

例:

```text
レビュー担当として、今回の変更を見て。
バグ、スマホ表示、学習難易度、ゲームバランスの観点で指摘して。
```

## 実装するときのルール

- 1回の変更では、1つのテーマだけ扱う
- 同時に複数エリアを増やさない
- 最初はカフェエリアを完成させる
- コード編集担当は1つにする
- 企画チャットではコードを触らない
- 実装前に「採用する仕様」を箇条書きで確定する
- push前に `node --check app.js` を実行する
- HTMLファイルのタグ構造が崩れていないか確認する
- スマホ幅（375px想定）でレイアウトが崩れていないかをコメントまたはdry-runで確認する
- GitHub Pages公開URL（前述のプロジェクト概要に記載）でページが開くか確認する

## 今の優先順位

### 今フェーズ（カフェ完成）

1. カフェエリアを楽しくする
2. 英単語・英文・英会話の質を上げる
3. 正解後の解説と復習を強くする
4. バリスタやカフェ内装の成長を分かりやすくする
5. スマホUIを整える

### 次フェーズ（エリア拡張）

6. ホテルエリアを追加する
7. 空港エリアを追加する
8. 夫婦共有ルームを検討する

## まだやらないこと

- 最初からオンライン同期を作り込む
- 最初からAI英会話APIを組み込む
- 複数エリアを一気に増やす
- キャラ画像生成に合わせてUIを大きく作り直す
- 対戦機能を先に作る

## 1機能の完了条件

1つの機能は、次を満たしたら完了とする。

- 何をする機能か説明できる
- スマホで使える
- 英語学習とゲーム報酬がつながっている
- XPや保存データが壊れない
- GitHub Pagesに反映済み
- 次に何を改善するか分かっている
